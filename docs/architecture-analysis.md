# Архитектура проекта и логика приложения

## 1. Что это за проект

Проект представляет собой консольное приложение на TypeScript, которое отслеживает расхождение цен между спотовыми и фьючерсными инструментами биржи Bybit. Основная идея — находить символы, для которых цена фьючерса и спота временно расходятся, а затем сигнализировать о потенциальной возможности для арбитража или статистического наблюдения за спредом.

Приложение работает в режиме реального времени: оно получает список символов, подписывается на обновления цены через WebSocket, вычисляет спред, сглаживает его и анализирует отклонение от исторической нормы.

---

## 2. Общая архитектура

Проект имеет модульную структуру с явным разделением ответственности:

- api — работа с внешними HTTP-источниками
- config — параметры и настройки поведения
- engine — оркестрация и бизнес-логика анализа
- indicators — математические вычисления и индикаторы
- ws — соединения по WebSocket
- types — общие типы

На уровне выполнения приложение запускается из точки входа [src/index.ts](src/index.ts), где последовательно выполняются следующие шаги:

1. Получение списка спот-символов
2. Получение списка фьючерс-символов
3. Формирование пересечения символов
4. Разделение на пакеты по 20 символов
5. Открытие WebSocket-подписок для каждого пакета

## 3. Структура пакетов и их роль

### 3.1 Пакет api

Путь: [src/api/symbols.ts](src/api/symbols.ts)

Задача пакета — получать информацию о доступных инструментах с API Bybit.

#### Содержимое

- getSpotSymbols()
- getFuturesSymbols()

#### Логика

- Используется HTTP-запрос к эндпоинту Bybit: /v5/market/instruments-info
- В зависимости от категории передается параметр category=spot или category=linear
- Из ответа берутся поля symbol и формируется Set<string>

#### Почему это важно

Этот модуль является стартовой точкой для построения universe. Без него приложение не знает, какие инструменты вообще доступны для мониторинга.

---

### 3.2 Пакет config

Путь: [src/config/index.ts](src/config/index.ts)

Содержит параметры анализа и фильтров.

#### Основные параметры

- historySize — длина истории для расчета статистики
- warmupSize — минимальная длина истории до начала анализа
- zThreshold — порог z-score для генерации сигнала
- minBasis — минимальный размер спреда, ниже которого сигнал игнорируется
- cooldownMs — защита от повторных сигналов в течение короткого периода
- emaAlpha — коэффициент сглаживания экспоненциальной средней

#### Роль

Этот модуль делает логику анализа настраиваемой. Вместо жёстко зашитых значений все ключевые параметры находятся в одном месте.

---

### 3.3 Пакет engine

Путь: [src/engine](src/engine)

Это ядро приложения. Здесь происходит объединение данных из сети, управление состоянием и запуск вычислений.

#### 3.3.1 universe.ts

Файл [src/engine/universe.ts](src/engine/universe.ts) строит набор символов, которые одновременно доступны и в споте, и в фьючерсах.

Логика очень простая:

- пройти по спот-набору
- проверить, есть ли такой же символ в наборе фьючерсов
- если есть — добавить в выходной список

Это создает базовый universe для анализа.

#### 3.3.2 scanner.ts

Файл [src/engine/scanner.ts](src/engine/scanner.ts) — главный аналитический модуль.

Он хранит состояние по каждому символу в памяти:

- текущая спотовая цена
- текущая фьючерсная цена
- история сглаженных значений
- EMA-значение

Когда приходит обновление цены, оно сохраняется в состояние и запускает функцию compute().

Основная логика compute():

1. Проверить, что обе цены известны
2. Рассчитать basis = (futures - spot) / spot
3. Отфильтровать слишком маленькие спреды
4. Сгладить спред через EMA
5. Добавить значение в историю
6. Если история недостаточно длинная — выйти
7. Рассчитать z-score относительно истории
8. При необходимости сгенерировать сигнал

### 3.4 Пакет indicators

Путь: [src/indicators](src/indicators)

#### 3.4.1 spread.ts

Файл [src/indicators/spread.ts](src/indicators/spread.ts) содержит чистые функции для расчета базового спреда и z-score.

- calculateBasis(futures, spot)
- calculateZScore(history, current)

Эти функции являются математической основой анализа.

#### 3.4.2 stats.ts

Файл [src/indicators/stats.ts](src/indicators/stats.ts) реализует базовую статистику:

- mean() — среднее значение
- std() — стандартное отклонение

Эти функции используются для расчета z-score.

---

### 3.5 Пакет ws

Путь: [src/ws/batch.ts](src/ws/batch.ts)

Этот пакет отвечает за сетевое соединение по WebSocket.

#### Логика

- Создается соединение к нужному endpoint для spot или futures
- При открытии подписывается на топики вида tickers.<symbol>
- При получении сообщения parses JSON
- Из topic извлекается символ
- Из payload берется lastPrice
- Вызывается callback onPrice(symbol, price)

#### Примечание

Подключение создается отдельно для каждого пакета символов. То есть приложение не использует один общий WebSocket на все символы, а создает несколько соединений.

---

### 3.6 Пакет types

Путь: [src/types/market.ts](src/types/market.ts)

Содержит тип PriceUpdate, который описывает структуру данных обновления цены.

Сегодня этот тип используется ограниченно, но он закладывает основу для более структурированного и расширяемого дизайна.

## 4. Поток выполнения приложения

### 4.1 Инициализация

В [src/index.ts](src/index.ts) стартует функция main().

Внутри выполняются следующие шаги:

- загрузка списка спот-символов
- загрузка списка фьючерс-символов
- пересечение этих множеств
- разбиение на чанки по 20 символов
- запуск подписок на spot/futures для каждого чанка

### 4.2 Подписка на данные

Для каждого чанка приложение запускает два WebSocket-соединения:

- одно для spot-потока
- одно для futures-потока

Это позволяет получать цены по нужным символам в реальном времени.

### 4.3 Обработка обновлений цен

При каждом обновлении цены вызывается:

- onSpot(symbol, price)
- onFutures(symbol, price)

Они обновляют состояние конкретного символа и вызывают compute().

### 4.4 Генерация сигнала

Если спред:

- достаточно велик,
- прошёл через EMA,
- достаточно стабилен статистически,
- не находится в кулдауне,

то приложение выводит сообщение в консоль.

## 5. Логика сигнального механизма

Основная идея сигнала основана на статистическом отклонении спреда от его собственной истории.

### 5.1 Базовый спред

$$
\text{basis} = \frac{\text{futures} - \text{spot}}{\text{spot}}
$$

Это относительное расхождение между ценой фьючерса и спота.

### 5.2 Сглаживание

Вместо использования сырого спреда применяется экспоненциальное сглаживание:

$$
\text{ema} = \alpha \cdot \text{basis} + (1 - \alpha) \cdot \text{ema}
$$

Это уменьшает шум и делает сигнал более устойчивым.

### 5.3 Z-score

Для оценки, насколько текущее значение отклоняется от среднего исторического уровня, используется z-score:

$$
z = \frac{\text{ema} - \mu}{\sigma}
$$

Если абсолютное значение $z$ превышает порог, сигнал считается сильным.

### 5.4 Фильтры

Перед генерацией сигнала применяются дополнительные фильтры:

- минимальный размер спреда
- необходимая длина истории
- кулдаун между сигналами

Это помогает уменьшить ложные срабатывания.

## 6. Сильные стороны текущей архитектуры

1. Модульность
   - каждый пакет отвечает за одну область ответственности

2. Простота
   - код легко читается и быстро понимается

3. Чистое разделение логики
   - математические функции отделены от сетевой логики

4. Настраиваемость
   - важные параметры вынесены в конфиг

5. Подходит для прототипа
   - приложение успешно выполняет базовую задачу мониторинга спреда

---

## 7. Ограничения текущей реализации

Хотя архитектура проста и понятна, она пока является прототипом. В ней нет:

- обработки ошибок и повторных подключений на уровне устойчивости
- сохранения состояния между перезапусками
- нормальной системы логирования
- тестов
- управления рисками и оркестрации сигналов
- интеграции с брокером или торговой системой

---

## 8. Краткое резюме по архитектуре

Проект построен как небольшой потоковый аналитический сервис. Его основная задача — получать цены по споту и фьючерсам, находить общие символы, вычислять относительный спред, сглаживать его и генерировать сигналы при статистическом отклонении.

С точки зрения структуры это хороший базовый skeleton для алгоритмического мониторинга, но для production-версии потребуется усиление устойчивости, наблюдаемости и функциональности.
