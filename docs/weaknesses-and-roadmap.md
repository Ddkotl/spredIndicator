# Недостатки проекта и дальнейшие шаги развития

## 1. Ключевые недостатки текущей реализации

### 1.1 Отсутствие устойчивости к сетевым сбоям

Сейчас приложение полагается на WebSocket-соединения без глубокой обработки сценариев обрыва связи. Если соединение закроется или будет нестабильно, часть данных может быть потеряна, а сигнализация станет неполной.

Что нужно улучшить:

- автоматическое переподключение
- backoff-повторные попытки
- восстановление подписок после reconnect
- обработка таймаутов и ошибок

---

### 1.2 Отсутствие долговременного состояния

Все данные хранятся только в памяти. При перезапуске приложения состояние теряется, а история спредов и сигналы обнуляются.

Это особенно критично для:

- анализа по времени
- обучения моделей
- повторного сравнения сигналов
- отладки поведения после рестарта

---

### 1.3 Простая логика сигналов

На данный момент сигнал строится на основе z-score и фильтров, но это довольно примитивный подход. Он может давать много ложных сигналов в волатильных условиях.

Что можно добавить:

- мультифакторную оценку
- проверку по нескольким таймфреймам
- корреляцию с объемом
- анализ волатильности
- фильтрацию по направлению движения рынка

---

### 1.4 Нет нормальной системы логирования

Сейчас приложение выводит сообщения в консоль. Это достаточно для прототипа, но недостаточно для полноценного продакшн-сервиса.

Нужны:

- структурированное логирование
- уровни логов
- запись в файлы или внешние системы
- метрики по сигналам, ошибкам и числу обновлений

---

### 1.5 Отсутствие тестов

В проекте нет автоматических тестов. Это делает сложным контроль качества при изменениях и повышает риск регрессий.

Следует добавить:

- unit-тесты для статистики и вычисления спреда
- тесты для логики фильтров
- тесты для сборки universe
- интеграционные тесты для обработчиков данных

---

### 1.6 Недостаточная структура данных

Хотя есть тип PriceUpdate, в проекте всё ещё много значений передается как примитивы: strings, numbers, booleans. Это ограничивает расширяемость.

Следует перейти к:

- более явным DTO и моделям состояния
- отдельным типам для сообщений WebSocket
- структурам для сигналов и событий

---

## 2. Практические улучшения, которые стоит сделать в первую очередь

### 2.1 Добавить слой устойчивости

Реализовать:

- reconnect для WebSocket
- retry на HTTP-запросы
- журналирование ошибок
- защиту от повторной обработки дублирующих сообщений

### 2.2 Ввести хранение состояния

Возможные варианты:

- локальная база данных SQLite
- PostgreSQL
- Redis
- JSONL-файлы для простого прототипа

Это позволит:

- хранить историю цен
- анализировать поведение по времени
- не терять данные после рестарта

### 2.3 Расширить логику сигналов

Сделать сигнал более “умным”:

- учитывать не только z-score, но и скорость изменения спреда
- включить проверку на экстремумы
- сравнивать текущий basis с несколькими окнами истории
- добавить оценку вероятности

### 2.4 Добавить метрики и наблюдаемость

Рекомендуется:

- считать число обновлений в секунду
- считать число сгенерированных сигналов
- отслеживать процент потерянных сообщений
- сохранять статистику по каждому символу

### 2.5 Покрыть код тестами

Начать с простых сценариев:

- расчет спреда
- расчет z-score при нулевом std
- фильтрация по минимальному basis
- cooldown между сигналами

---

## 3. Среднесрочные шаги развития

### 3.1 Версия 2.0: более зрелый аналитический движок

Включает:

- устойчивые WebSocket-клиенты
- хранение истории в базе данных
- нормальную архитектуру событий
- систему конфигурации через env
- логирование в структурированном формате

### 3.2 Версия 3.0: торговая логика

Если цель — не только анализ, но и автоматическая торговля, стоит добавить:

- управление позициями
- расчёт риска
- правила входа и выхода
- ограничение по капиталу
- защиту от резких просадок

### 3.3 Версия 4.0: машинное обучение и прогнозирование

Если проект будет развиваться как исследовательская платформа, можно:

- собирать больше признаков
- обучать модели на исторических данных
- прогнозировать направление спреда
- оценивать вероятность срабатывания сигнала

---

## 4. Возможные направления для продукта

### Вариант A — аналитический мониторинг

Фокус на наблюдении за спредами, их визуализации и анализе.

Подходит, если основная цель — исследование рынка.

### Вариант B — торговый сигнализатор

Фокус на генерации сигналов, управлении рисками и автоматизации.

Подходит, если цель — применять стратегию на реальном рынке.

### Вариант C — исследовательская платформа

Фокус на накоплении исторических данных, моделировании и обучении.

Подходит, если проект должен стать основой для ML-аналитики.

---

## 5. Рекомендованный порядок развития

1. Укрепить сетевую устойчивость
2. Добавить хранение состояния и истории
3. Улучшить систему сигналов
4. Добавить логирование и метрики
5. Покрыть проект тестами
6. При необходимости перейти к торговой логике

---

## 6. Итог

Проект уже показывает рабочую идею и обладает понятной архитектурой, но сейчас он находится на уровне прототипа. Его сильная сторона — простота и прозрачность. Однако для перехода к более серьёзному использованию потребуется усиление устойчивости, аналитической глубины и инфраструктуры наблюдаемости.

Если развивать проект дальше, лучше делать это пошагово: сначала — надёжность, затем — качество сигналов, затем — инфраструктура и торговая логика.
